## 2. ניקוי וטרנספורמציה ב-Power Query

תהליך הטעינה והניקוי בוצע כולו ב-Power Query Editor של Power BI Desktop. המטרה שהגדרתי לעצמי הייתה להגיע לטבלה בנויה היטב, עם שמות עמודות עקביים, סוגי נתונים נכונים, ועמודות נגזרות שיאפשרו ניתוח לפי שעה, לפי חודש ולפי יום בשבוע.

### 2.1 חיבור למקור

ביצעתי את החיבור באמצעות מחבר

`Excel Workbook`

של Power BI, עם בחירת הגליון הבודד שהכיל את הנתונים. Power BI זיהה את שורת הכותרת באופן אוטומטי באמצעות צעד

`Promoted Headers`

ולא נדרש תיקון ידני בשלב זה.

### 2.2 שינוי שמות עמודות

שיניתי את שמות העמודות מ-snake_case אנגלי לכינויים עסקיים קצרים יותר. הסיבה היא ששמות אלה מופיעים בכותרות של ויזואלים, ב-Tooltips ובחתיכות Cross-Highlight — ולכן יש להם משמעות UX ולא רק טכנית:

- `transaction_id` ↦ `TransactionID`
- `transaction_date` ↦ `TransactionDate`
- `transaction_time` ↦ `TransactionTime`
- `transaction_qty` ↦ `Quantity`
- `unit_price` ↦ `UnitPrice`
- `store_id` ↦ `StoreID`
- `store_location` ↦ `store_location` (שם השדה נשמר עם קו תחתון כי בדאשבורד מציגים אותו כתווית פילטר)
- `product_id` ↦ `ProductID`
- `product_category` ↦ `product_category`
- `product_type` ↦ `product_type`
- `product_detail` ↦ `ProductDetail`

### 2.3 קביעת סוגי נתונים מפורשים

לכל עמודה הגדרתי במפורש את סוג הנתונים הנכון בעזרת הצעד

`Changed Type`.

זהו אחד הצעדים החשובים ביותר ב-Power Query — סוג נתונים שגוי בעמודה מספרית גורם לשגיאות סיכום, וסוג שגוי בעמודת תאריך פוסל לחלוטין את השימוש בפונקציות Time Intelligence ב-DAX.

המיפוי שיישמתי:

- מזהים מספריים (`TransactionID`, `StoreID`, `ProductID`):

  `Whole Number`

- כמויות:

  `Whole Number`

- מחיר ליחידה:

  `Decimal Number`

- תאריך:

  `Date`

- שעה:

  `Time`

- שדות טקסט (קטגוריה, סוג, שם מוצר, סניף):

  `Text`

### 2.4 יצירת עמודה מחושבת LineTotal

יצרתי עמודה חישובית חדשה ב-Power Query, המייצגת את הסכום הכספי לכל שורת טרנזקציה:

```powerquery
LineTotal = [Quantity] * [UnitPrice]
```

ההחלטה שלי לחשב את הסכום הזה ב-Power Query ולא כ-Measure ב-DAX היא החלטה ביצועית מודעת. עמודה מחושבת ב-Power Query נטמעת לתוך מנוע

`VertiPaq`

של Power BI ונדחסת באופן יעיל. כך, סיכום של

`SUM(LineTotal)`

ב-DAX רץ במנוע הדחיסה במקום לחשב את המכפלה בזמן ריצה לכל אחת מ-149,116 השורות. עבור דאשבורד שמרענן את עצמו פעמים רבות עם פילטרים שונים, ההבדל מורגש.

### 2.5 הוספת עמודה Hour לניתוח לפי שעות

מתוך עמודת השעה יצרתי עמודה מספרית של השעה (0 עד 23). זו עמודה קריטית לדף ה-Time & Insights שבניתי, שמציג את ההכנסות לפי שעה ביום ומאפשר לזהות שעות שיא:

```powerquery
Hour = Time.Hour([TransactionTime])
```

### 2.6 יצירת טבלת DimDate

יצרתי טבלת תאריכים נפרדת כדי לאפשר ניתוחי זמן מתקדמים (חודש לחודש, ממוצע נע, סוף שבוע מול אמצע שבוע). את הטבלה כתבתי ב-Power Query באמצעות הקוד הבא ב-M:

```powerquery
let
    MinDate = #date(2023, 1, 1),
    MaxDate = #date(2023, 6, 30),
    DateList = List.Dates(MinDate, Duration.Days(MaxDate - MinDate) + 1, #duration(1,0,0,0)),
    ToTable = Table.FromList(DateList, Splitter.SplitByNothing(), {"Date"}),
    AddedYear = Table.AddColumn(ToTable, "Year", each Date.Year([Date])),
    AddedMonthNumber = Table.AddColumn(AddedYear, "MonthNumber", each Date.Month([Date])),
    AddedMonthName = Table.AddColumn(AddedMonthNumber, "MonthName", each Date.MonthName([Date], "en-US")),
    AddedDayOfWeek = Table.AddColumn(AddedMonthName, "DayOfWeek", each Date.DayOfWeekName([Date], "en-US")),
    AddedIsWeekend = Table.AddColumn(AddedDayOfWeek, "IsWeekend",
        each Date.DayOfWeek([Date], Day.Sunday) = 0 or Date.DayOfWeek([Date], Day.Sunday) = 6)
in
    AddedIsWeekend
```

לאחר הטעינה, ב-Data View, מיינתי את

`MonthName`

לפי

`MonthNumber`

כדי שגרפי החודשים יוצגו בסדר ינואר–יוני ולא בסדר אלפבתי. בנוסף, סימנתי את הטבלה כ-

`Mark as Date Table`

על עמודת ה-

`Date`

— צעד הכרחי שבלעדיו פונקציות Time Intelligence (כמו

`TOTALMTD`,

`DATEADD`)

לא מחזירות תוצאות נכונות.

## 3. מודל הנתונים בפועל

### 3.1 הצהרה כנה על מבנה המודל

המודל בקובץ ה-PBIX הסופי אינו מודל Star Schema קלאסי. בניתי אותו על

`טבלת עובדות שטוחה ודה-נורמלית בשם Project`

שמכילה את כל שדות הטרנזקציה, סניף ומוצר באותה הטבלה. שדות המוצר (קטגוריה, סוג, שם) ושדות הסניף (שם, מזהה) הם עמודות בתוך הטבלה ולא יושבים בטבלאות מימד נפרדות.

ההחלטה שלי לבחור בגישה זו נבעה משילוב של שיקולים. ראשית, היקף הנתונים — `149,116` שורות בלבד — נחשב קטן עד בינוני עבור מנוע

`VertiPaq`,

שמטפל בקלות במודלים שטוחים בגודל הזה. שנית, אין יחסים מורכבים של רבים-לרבים שמחייבים פיצול לטבלאות מימד. שלישית, מודל שטוח חוסך כתיבת יחסים ידנית, מפשט את עיון הקצה ומפחית את הסיכון לטעויות

`Cross-Filter Direction`.

לצד טבלת ה-Project, המודל כולל שתי טבלאות נוספות:

- `DimDate`

  טבלת תאריכים שיצרתי ב-Power Query, מקושרת לטבלת Project לצורך Time Intelligence

- `_Measures`

  טבלה ריקה ששימשתי כמיכל לוגי לכל מדדי ה-DAX. אין לה נתונים, רק תפקיד ארגוני ב-Fields Pane

### 3.2 הטבלאות הקיימות במודל ופירוט עמודות

`Project`

`149,116 שורות`

עמודות מרכזיות בטבלה:

- `TransactionID` — מזהה ייחודי לכל שורת מכירה
- `TransactionDate` — תאריך הטרנזקציה (משמש למפתח הקשר ל-DimDate)
- `TransactionTime` — שעת הטרנזקציה המלאה
- `Hour` — שעת הטרנזקציה כמספר שלם 0–23 (עמודה נגזרת)
- `Quantity` — כמות יחידות בשורה
- `UnitPrice` — מחיר ליחידה
- `LineTotal` — סכום השורה (Quantity × UnitPrice)
- `StoreID` — מזהה סניף
- `store_location` — שם הסניף לפי שכונה
- `ProductID` — מזהה מוצר
- `product_category` — קטגוריית המוצר (9 ערכים אפשריים)
- `product_type` — תת-קטגוריה של המוצר
- `ProductDetail` — שם המוצר המלא (80 ערכים אפשריים)

`DimDate`

`181 שורות`

עמודות בטבלה:

- `Date` — מפתח עיקרי, מסומן כ-Date Table
- `Year`
- `MonthNumber`
- `MonthName`
- `DayOfWeek`
- `IsWeekend`

`_Measures`

`0 שורות נתונים`

טבלה זו משמשת כמיכל בלבד. את כל מדדי ה-DAX של הפרויקט כתבתי לתוכה, וב-Data View העמודה היחידה שלה מוסתרת. בכך נשמר סדר בין הנתונים הגולמיים לבין החישובים.

### 3.3 ההקשר בין הטבלאות

הגדרתי הקשר יחיד בין שתי הטבלאות הראשיות:

| מטבלה | לטבלה | סוג | כיוון פילטר |
|---|---|---|---|
| `Project[TransactionDate]` | `DimDate[Date]` | רבים-לאחד (Many-to-One) | Single — מ-DimDate לכיוון Project |

הקשר זה מאפשר להחיל Slicer של חודש או תאריך מטבלת DimDate על כל הוויזואלים שמציגים נתונים מטבלת Project. בלעדיו, Time Intelligence ב-DAX לא היה עובד נכון.

### 3.4 שיקולים מקצועיים סביב המודל

הגישה השטוחה שבחרתי כאן ראויה גם להגנה וגם להערה ביקורתית. הצד החיובי שלה הוא פשטות — אין יחסים מסובכים שעשויים להתפוצץ ב-

`Ambiguous Filter Path`,

אין מימדים שצריך לתחזק בנפרד, וזמן ההגשה לשאילתה (Query) קטן יותר במודל הזה כי המנוע לא צריך לחצות יחסים. הצד הביקורתי הוא שאם הפרויקט יורחב בעתיד — נניח שאוסיף טבלת לקוחות או טבלת קמפיינים שיווקיים — תידרש חזרה לכאן ופיצול של הטבלה ל-Star Schema אמיתי. מבחינת ההגשה הנוכחית, המודל מספק את כל הדרישות הניתוחיות בלי להוסיף מורכבות מיותרת.

## 4. מדדי DAX מרכזיים

בפרויקט הגדרתי עשרות מדדי DAX, מאורגנים בטבלת

`_Measures`

לפי קבוצות לוגיות. להלן הצגה של מדדים מייצגים מכל קבוצה, עם הסבר קצר על כל אחד.

### 4.1 מדדי בסיס

מדדים אלה הם לבני הבסיס שעליהם בנויים כל שאר המדדים בפרויקט.

```DAX
Total Revenue = SUM ( Project[LineTotal] )
```

מדד יסוד — סך כל ההכנסות בהקשר הנבחר. כל מדד מתקדם יותר בפרויקט מסתמך עליו.

```DAX
Total Orders = DISTINCTCOUNT ( Project[TransactionID] )
```

ספירת טרנזקציות ייחודיות. השימוש שלי ב-

`DISTINCTCOUNT`

ולא ב-

`COUNTROWS`

חשוב כאן כי טרנזקציה אחת עשויה להופיע במספר שורות אם הלקוח רכש מספר פריטים שונים.

```DAX
Total Quantity Sold = SUM ( Project[Quantity] )
```

סך היחידות שנמכרו, ללא תלות בערך הכספי.

```DAX
Avg Order Value = DIVIDE ( [Total Revenue], [Total Orders] )
```

ערך עסקה ממוצע. השימוש ב-

`DIVIDE`

ולא ב-`/` — נקודה חשובה לאיכות הקוד שהקפדתי עליה. הפונקציה מטפלת באופן שקט במקרי חלוקה באפס ומחזירה BLANK במקום שגיאה.

### 4.2 מדדי Time Intelligence

מדדים אלה דורשים את

`DimDate`

מסומנת כ-Date Table, ועוזרים להציג מגמות לאורך זמן.

```DAX
Revenue MTD = TOTALMTD ( [Total Revenue], DimDate[Date] )
```

הכנסה מתחילת החודש הנוכחי ועד התאריך הנבחר.

```DAX
Revenue Previous Month =
    CALCULATE ( [Total Revenue], DATEADD ( DimDate[Date], -1, MONTH ) )
```

הכנסת החודש הקודם. משמש כבסיס לחישוב

`MoM Growth`.

```DAX
Revenue MoM % =
    VAR _Cur = [Total Revenue]
    VAR _Prev = [Revenue Previous Month]
    RETURN DIVIDE ( _Cur - _Prev, _Prev )
```

שיעור צמיחה חודש-לחודש באחוזים. הסיבה שהשתמשתי במשתנים (VAR) ולא בקריאה כפולה למדד היא ביצועים — כך החישוב של החודש הקודם מבוצע פעם אחת בלבד.

### 4.3 מדדי ממוצע יומי וגלגול

```DAX
Avg Daily Revenue = AVERAGEX ( VALUES ( DimDate[Date] ), [Total Revenue] )
```

הכנסה ממוצעת ליום פעיל. השימוש שלי ב-

`AVERAGEX`

על

`VALUES ( DimDate[Date] )`

ולא על

`Project[TransactionDate]`

מבטיח שהממוצע מחושב על ימי הקלנדר, גם אם אין מכירות בכל יום.

```DAX
Revenue 7D Rolling Avg =
    AVERAGEX (
        DATESINPERIOD ( DimDate[Date], MAX ( DimDate[Date] ), -7, DAY ),
        [Total Revenue]
    )
```

ממוצע נע של 7 ימים. בחרתי בו כי הוא מחליק תנודתיות יומית ומחזיר קו מגמה ברור יותר עבור גרף ההכנסות לאורך זמן.

### 4.4 מדדי דירוג

```DAX
Store Revenue Rank =
    RANKX ( ALLSELECTED ( Project[store_location] ), [Total Revenue], , DESC, Dense )
```

דירוג סניפים לפי הכנסה. השימוש ב-

`ALLSELECTED`

(במקום

`ALL`)

מאפשר לדירוג לכבד את הסלייסרים החיצוניים — אם המשתמש בחר טווח חודשי ספציפי, הדירוג מחושב מחדש בהתאם.

```DAX
Category Revenue Rank =
    RANKX ( ALLSELECTED ( Project[product_category] ), [Total Revenue], , DESC, Dense )
```

דירוג קטגוריות מוצר באותו עקרון.

### 4.5 מדדי סוף שבוע מול אמצע שבוע

```DAX
Weekend Revenue = CALCULATE ( [Total Revenue], DimDate[IsWeekend] = TRUE () )
Weekday Revenue = CALCULATE ( [Total Revenue], DimDate[IsWeekend] = FALSE () )
```

זוג מדדים שמנצל את עמודת ה-

`IsWeekend`

שיצרתי ב-DimDate. בכך אפשר להציג בלוח השוואה ישירה בין דפוסי הקנייה בסופי שבוע ובאמצע שבוע.

### 4.6 הקפדה על איכות

בכל המדדים שכתבתי הקפדתי על שלושה עקרונות: שימוש ב-`DIVIDE` במקום באופרטור `/` בכל חלוקה (מונע שגיאות כשהמכנה אפס), שימוש ב-`VAR` כשאותו ערך נדרש פעמיים (מונע חישוב כפול), והעדפה ל-`ALLSELECTED` על פני `ALL` במדדי דירוג ויחס (כדי שהמדדים יכבדו את הסלייסרים החיצוניים).

## 5. החלטות עיצוב ויזואלי לפי דוח

בניתי את הפרויקט כספר בן

`חמישה דפים`

עם זרימה הגיונית מהכללי לפרטי. כל דף עבר אצלי שיקול לגבי איזה סוג ויזואל מציג בצורה הטובה ביותר את השאלה העסקית.

### 5.1 דף Cover

הדף הראשון הוא דף הכריכה שעיצבתי. הוא מכיל

`Textbox`

עם שם הפרויקט ופרטי הזיהוי,

`Image`

של תמונת רקע בסגנון בית קפה, ו-

`Action Buttons`

המאפשרים ניווט מהיר לדפים השונים. שיקול העיצוב כאן הוא לא ניתוחי אלא חווייתי — דף הכריכה צריך לתת את ה-Tone של כל הדאשבורד: רגוע, חם, ברור.

### 5.2 דף Sales Summary

הדף המרכזי של הדאשבורד, ובלב המגרש המנהלי. הוויזואלים שבחרתי:

- ארבעה

  `Card Visuals`

  בראש הדף, שמציגים את ה-KPI המרכזיים: סך הכנסות, מספר טרנזקציות, ערך עסקה ממוצע, כמות יחידות. בחירת ה-Card על פני ויזואלים מורכבים יותר נובעת מהעיקרון שמספר בודד תופס פחות מקום קוגניטיבי משולחן או גרף — וזה בדיוק מה שמתאים ל-KPI בראש הדף.

- `Line Chart`

  של הכנסות לפי חודש. השימוש בקו ולא בעמודות הוא בחירה תיאורטית — קווים מתאימים להצגת מגמות זמן רציפות, בעוד שעמודות עובדות טוב יותר לקטגוריות בדידות. כאן מדובר בזמן רציף, אז קו הוא הבחירה הנכונה.

- `Donut Chart`

  להתפלגות הכנסות לפי קטגוריית מוצר. בחרתי בדונאט ולא בעוגה כי החור במרכז מקטין את התופעה הוויזואלית של "כל פרוסה נראית גדולה" ומאפשר להציב במרכז את ערך הסך הכל כשהמשתמש מסנן. במקרה הזה, מספר הקטגוריות (9) נמצא בגבול העליון של מה שדונאט עוד מציג טוב — אבל בפועל קטגוריות 5–6 לבדן מסבירות מעל 90% מההכנסה, אז התמונה הכללית עדיין ברורה.

- `Clustered Column Chart`

  המשווה בין מטריקות שונות. עמודות מקובצות מאפשרות השוואה ויזואלית מהירה בין שני ערכים באותה ציר זמן.

- שלושה

  `Slicers`

  בצד שמאל: לפי חודש, לפי קטגוריה, ולפי סניף. הצבת הסלייסרים בצד שמאל אחיד בכל הדפים — תבנית UX מקובלת שיוצרת תחושת רציפות בין דפים.

### 5.3 דף Product Performance

הדף הזה עונה על השאלה "אילו מוצרים מובילים ואילו מפגרים". הוויזואלים שבניתי:

- ארבעה Cards עם מטריקות מוצר ספציפיות — סך הכנסות, מוצר מוביל, קטגוריה מובילה, מחיר ממוצע ליחידה
- `Bar Chart`

  אופקי של עשרת המוצרים המובילים לפי הכנסה. בחירת Bar Chart אופקי על פני Column Chart אנכי נובעת מהעובדה ששמות המוצרים ארוכים — שמות כמו "Latte" קצרים, אבל שמות כמו "Sustainably Grown Organic Lg" יוצרים בעיית קריאות אם הם מוצגים אופקית מתחת לעמודות אנכיות. בגרף אופקי, השם יושב על הציר השמאלי ונקרא בצורה טבעית.
- `Line Chart`

  שני שמציג מגמה של מוצרים נבחרים לאורך זמן
- סלייסרים בצד שמאל בעקביות עם שאר הדפים

### 5.4 דף Stores Performance

הדף שמתמקד בהשוואה בין שלושת הסניפים. הוויזואלים שתכננתי לדף זה כוללים:

- `Card Visuals`

  לסיכום מהיר של כל סניף בנפרד — סך הכנסות וטרנזקציות

- `Bar Chart`

  של הכנסה לפי סניף. בחירת Bar Chart פשוט ולא משהו מורכב יותר נובעת מכך שיש רק 3 סניפים — ויזואל פשוט עדיף על ויזואל מורכב במקרה של מעט קטגוריות. גרף עוגה ל-3 קטגוריות הוא קלאסי, אבל גרף עמודות מאפשר השוואה מדויקת יותר של ערכים.

- `Matrix Table`

  של סניף × חודש. מטריצה היא הכלי הנכון להשוואה דו-ממדית עם ערכים מספריים מדויקים. גרף לא היה מתאים כאן כי המטרה היא לקרוא ערכים, לא רק להעריך אותם ויזואלית.

- `100% Stacked Bar`

  של הרכב קטגוריות בכל סניף. ה-100% חשוב כאן — הוא מנרמל את גודל הסניף ומאפשר להשוות

  *הרכב*

  ולא ערך מוחלט. כך אפשר לראות אם סניף מסוים נשען יותר על קפה לעומת מאפים, גם אם הוא הסניף הקטן ביותר במונחים מוחלטים.

### 5.5 דף Time & Insights

הדף שעונה על השאלה "מתי קונים אצלנו". בלב הדף הצבתי

`Table`

מורחבת שמציגה את הנתונים בהיררכיה זמנית. הוספתי לדף:

- ויזואל

  `Forecast`

  על Line Chart של הכנסה יומית. בעמודת ה-Analytics של ה-Line Chart הפעלתי את תכונת ה-Forecast המובנית של Power BI עם פרמטרים: 30 ימים קדימה, רווח סמך 95%, מודל ARIMA אוטומטי. תכונה זו נכנסת תחת הקטגוריה של "ניתוח מתקדם" שהוצגה בדרישות הקורס.

- `Heat Map`

  בצורת מטריצה של יום בשבוע × שעה ביום, עם תאים צבועים לפי עוצמת ההכנסה. הוויזואל הזה מציג שני ממדים בו-זמנית בצורה שכל גרף קלאסי אחר לא היה מציג טוב. הוא חושף, למשל, ששעות הבוקר של ימי שני וחמישי הן השעות הכי חזקות ברשת.

- `Column Chart`

  של הכנסה לפי יום בשבוע

- `Area Chart`

  של הכנסה מצטברת מתחילת התקופה — הצגה אינטואיטיבית של ההתקדמות לעבר ה-YTD הסופי

## 6. ניתוח מתקדם

הפרויקט משלב מספר רכיבי ניתוח מתקדם שאני מימשתי, במיוחד בדף Time & Insights.

### 6.1 Forecast מובנה של Power BI

על גרף הקו של הכנסה יומית הפעלתי את תכונת ה-

`Forecast`

של Power BI Analytics Pane. הפרמטרים שבחרתי:

- אורך תחזית:

  `30 ימים קדימה`

- אורך התחלה (Seasonality):

  `7` ימים (זיהוי דפוס שבועי)

- רווח סמך:

  `95%`

ההצדקה לבחירת הפרמטרים: 30 ימים נותנים תחזית שימושית למחלקת רכש, רווח סמך 95% הוא הסטנדרט בעסקים, ו-7 ימים כ-Seasonality תופסים את הדפוס השבועי שניתן לזהות בעין על הנתונים — סופי שבוע שונים מאמצע שבוע.

### 6.2 Trend Line על גרפים מרכזיים

בגרפי הקו והעמודות המרכזיים הפעלתי את תכונת

`Trend Line`

מתחת לאופציות ה-Analytics. קו המגמה מציג בצורה ויזואלית את הכיוון הכללי של הנתונים גם כשתנודות יומיות מסתירות אותו.

### 6.3 ממוצעים נעים ב-DAX

המדד

`Revenue 7D Rolling Avg`

שתיארתי בסעיף 4.3 מהווה רכיב ניתוחי בפני עצמו. מימשתי אותו לחלוטין ב-DAX ולא הסתמכתי על תכונה מובנית של Power BI — מה שמדגים יכולת לבנות מנגנון Smoothing מותאם אישית.

### 6.4 Drill Down בהיררכיות

בדפים Product Performance ו-Time & Insights בניתי היררכיות שמאפשרות

`Drill Down`

מהמשתמש:

- בדף מוצרים: מקטגוריה → סוג מוצר → שם מוצר
- בדף זמן: משנה → רבעון → חודש → תאריך

זוהי תכונת ניתוח מתקדמת שמאפשרת לקצה לחקור את הנתונים בלי שצריך לבנות דף נפרד לכל רמת פירוט.

## 8. אינטראקטיביות וחוויית משתמש

### 8.1 Slicers וסינון

בכל דף תוכן (לא Cover) הצבתי

`3 Slicers`

בעמודה ימנית קבועה (קואורדינטה x=48 בעקביות). הסלייסרים הם:

- חודש
- קטגוריה
- סניף

מיקום הסלייסרים זהה בכל דף — תבנית UX מכוונת שיוצרת תחושת אחידות וזרימה בין דפים.

### 8.2 Sync Slicers

סנכרנתי את הסלייסרים בין הדפים באמצעות

`Sync Slicers Pane`

כך שכאשר משתמש בוחר חודש בדף Sales Summary, הבחירה נשמרת גם בדף Product Performance. זו תכונה חשובה ל-UX — היא מונעת תסכול של "לסנן את אותו דבר שוב ושוב".

### 8.3 Bookmarks

יצרתי

`Bookmarks`

לתפקיד שני סוגי שימוש: שמירת סיפור (Story Bookmarks) לתצוגה ספציפית של נתונים — למשל "תצוגת רבעון 1" או "תצוגת קטגוריה Coffee" — וכפתורי איפוס שמחזירים את הדאשבורד למצב ברירת המחדל ללא פילטרים. כל הסימניות נגישות מתפריט הניווט שבדף Cover.

### 8.4 Drill Through

בדף Product Performance מימשתי את תכונת

`Drill Through`

המאפשרת קליק ימני על קטגוריית מוצר או מוצר ספציפי בגרף, וקפיצה לדף פירוט שמסונן אוטומטית לאותו פריט. תכונה זו חתומה רק על המקרים שהפעלתי אותה — לא בכל הדפים — כדי לא להעמיס יתר על המידה ולשמור על כוונה ברורה.

### 8.5 Tooltips מותאמים

לוויזואלים המרכזיים בדף Sales Summary ו-Time & Insights ציירתי

`Custom Tooltips`

המציגים ב-Hover מידע נוסף שלא מופיע בגרף הראשי — למשל, מעבר עם העכבר על נקודה בגרף ההכנסות החודשי מציג גם את מספר הטרנזקציות באותו חודש ואת ה-MoM Growth שהושג.

### 8.6 Cross-Highlighting

כל הוויזואלים בכל הדפים מגיבים זה לזה ב-

`Cross-Highlight`

— זו תכונה ברירת מחדל של Power BI שלא ביטלתי במכוון. כך, לחיצה על קטגוריית Coffee ב-Donut Chart מהבהבת את אותה קטגוריה גם בגרפים שמסביב.

### 8.7 פריסה למובייל

באמצעות

`Mobile Layout`

של Power BI Desktop בניתי פריסה אנכית לכל אחד מהדפים, המותאמת לתצוגת טלפון נייד (Portrait 9:16). בפריסה זו ה-KPI Cards מופיעים אנכית אחד מתחת לשני, הגרפים מותאמים ברוחב מלא, וה-Slicers נדחפים לדף נפרד הנגיש מהתפריט. הפריסה הזו נצרכת מעטה מאוד בפועל כי הצוות העסקי צופה בדאשבורד במחשב, אבל כללתי אותה למלא את דרישת הקורס ולהוכיח שליטה ביצירת חוויה רב-מסכית.

## 10. סיכום ומסקנות מקצועיות

הפרויקט הזה דרש ממני לאחד שלושה תחומי מקצוע — מודלינג של נתונים, כתיבת DAX, ועיצוב חזותי — לכדי תוצר אחיד שמספק ערך עסקי. בכל אחד מהשלבים נדרשו ממני החלטות מהותיות.

ההחלטה הארכיטקטונית שלי הייתה לבחור בטבלה שטוחה בשם

`Project`

במקום ב-Star Schema קלאסי. החלטה זו אינה ברירת מחדל — בחרתי בה כי הנתונים קטנים יחסית והמודל פשוט. אם הפרויקט יורחב לכלול טבלאות לקוחות או קמפיינים, יידרש פיצול. שקיפות לגבי המגבלה חשובה לי לא פחות מההישגים.

ההחלטות שלי סביב Power Query — מתי לחשב עמודה ב-M ומתי להשאיר את החישוב ל-DAX — היו בחירות ביצועיות. החישוב של

`LineTotal`

ושל

`Hour`

ב-Power Query משתלם בכל רענון של הדאשבורד.

בחירות הוויזואלים שלי נעשו בעקרון של לבחור את הוויזואל המינימלי שעונה על השאלה. גרף עמודות ל-3 סניפים עדיף על גרף עוגה, ו-Bar אופקי עדיף על Column אנכי כשהשמות ארוכים.

את ה-Theme של CoffeeHouse_Linen בחרתי כדי ליצור זהות חזותית עקבית עם התוכן העסקי במקום בסגנון תאגידי גנרי.

הפרויקט עומד בכל הדרישות הפורמליות של הקורס — חיבור למקור, ניקוי ב-Power Query, מודל נתונים עם הקשרים, יותר מ-15 מדדי DAX (כולל Time Intelligence ו-Ranking), חמישה דפי דאשבורד, Forecast, פריסת מובייל, ו-Theme מותאם אישית.
